# План тестирования BookBox API

## 1. Цели тестирования

### 1.1 Функциональное тестирование
**Цель:** подтвердить корректную работу точек `/register` и `/login`, а также статической выдачи фронтенда.

**Что проверяется:**
- Регистрация пользователя (POST /register)
- Обработка дубликатов логинов
- Авторизация пользователя (POST /login)
- Обработка неверных учетных данных
- Валидация обязательных полей
- Запуск и доступность статических страниц (`index.html`, `index1.html`, `index2.html`)

### 1.2 Пользовательское тестирование
**Цель:** оценить удобство взаимодействия с формами фронтенда и простоту REST-запросов через Postman/Swagger mock.

**Что проверяется:**
- Понятность UI (навигация, кнопки `BUY`)
- Ясность сообщений об ошибках при регистрации/входе
- Простота выполнения REST-запросов через Postman
- Предложения по улучшению каталога и футера

---

## 2. Инструменты и среда тестирования

### 2.1 Инструменты
- **Postman 11** — ручные REST-запросы
- **curl** — быстрая проверка из терминала
- **VS Code REST Client** — сохраненные запросы
- **SQLite Browser** — просмотр содержимого `database.db`
- **Google Forms** — сбор пользовательской обратной связи

### 2.2 Среда тестирования
- **ОС:** Windows 10/11
- **Node.js:** 20.x
- **Сервер:** Express (локально, http://127.0.0.1:3000)
- **Браузер:** Chrome/Edge (для UI страниц)

### 2.3 Подготовка к тестированию
1. `npm install`
2. `node server.js`
3. Убедиться, что сервер слушает порт 3000
4. Открыть `http://127.0.0.1:3000/index1.html`

---

## 3. Типы тестирования

### 3.1 Позитивное
- Регистрация с валидными логином/паролем
- Вход существующего пользователя
- Отображение главной страницы

### 3.2 Негативное
- Регистрация с уже занятым логином
- Пустые поля / слишком короткий пароль
- Вход с неверным паролем

### 3.3 Граничное
- Минимальная длина логина (3 символа)
- Максимальная длина логина (50 символов)
- Пароль из спецсимволов
- Массовые запросы (rate sanity)

### 3.4 Интеграционное
- Регистрация → автоматический вход
- Работа UI форм с REST-конечной точкой (через fetch)
- Проверка, что база получает записи, а статика отдаётся корректно

---

## 4. Критерии приемки

### 4.1 Функциональные
- Все заявленные эндпоинты возвращают ожидаемые ответы
- Повторная регистрация отдаёт понятную ошибку
- Авторизация доступна только для валидных пары логин/пароль
- Статика загружается без 404 и ошибок в консоли

### 4.2 Пользовательские
- Тестировщики оценивают удобство UI ≥ 4/5
- Сообщения об ошибках понятны
- Отчёт по тестированию содержит ≥ 90% успешных кейсов

---

## 5. План выполнения
1. **Подготовка** (0.5 дня) – настройка среды, очистка БД
2. **Функциональное тестирование** (1 день) – полный прогон REST-кейсов
3. **UI тестирование** (0.5 дня) – проверка страниц, навигации и форм
4. **Пользовательское тестирование** (1 день) – сессии с одногруппниками + Google Forms
5. **Анализ и отчётность** (0.5 дня) – фиксация результатов, обновление `issues.md`

---

## 6. Риски и ограничения
- Нет хеширования паролей → риски безопасности (известно)
- Отсутствие токенов → невозможно протестировать защищённые маршруты
- Локальная среда, возможны отличия от продакшена
- Ограниченное число тестировщиков (2–3 человека)

---

## 7. Метрики успеха
- Покрытие тестами: 100% текущих эндпоинтов
- Успешность кейсов: ≥ 90%
- Количество найденных дефектов задокументировано в `issues.md`
- Получено ≥ 4 ответа в Google Forms
- Средняя пользовательская оценка ≥ 4.0/5.0
